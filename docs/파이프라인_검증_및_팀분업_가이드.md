## 1. 엔드투엔드 파이프라인 검증

### 1.1 검증 결론: 정상 동작

`./run.sh all` 한 번으로 전체 파이프라인이 순서대로 실행됩니다. 각 단계가 이전 단계의 산출물을 자동으로 찾고, 없으면 해당 단계를 내부적으로 호출하는 **자기 복구(self-healing)** 구조입니다.

### 1.2 단계별 의존성 체인

```
data → vader → eda → [tune] → benchmark → freeze-study → xai → dashboard
```

| 단계 | 입력 의존성 | 출력 산출물 | 자동 복구 | 비고 |
|------|-----------|-----------|:---------:|------|
| **data** | 없음 (최초 단계) | `data_splits.pkl`, `experiment_config.json`, `data_profile.json` | - | HateXplain 자동 다운로드 |
| **vader** | `data_splits.pkl` | `vader_features.pkl`, `vader_*.csv` | `prepare_data()` 자동 호출 | 캐시 존재 시 스킵 |
| **eda** | `data_splits.pkl` + `vader_features.pkl` | `eda_summary.json`, 시각화 5종 | data + vader 모두 자동 호출 | 모델 학습 전 필수 |
| **tune** | `data_splits.pkl` + `vader_features.pkl` | `transformer_tuning_best.json` | data + vader 자동 호출 | **선택 단계** |
| **benchmark** | data + vader + tune(선택) | `benchmark_summary.csv`, `best_models.json`, `significance_tests.csv` | tune 결과 없으면 기본값 사용 | 7개 모델 구성 |
| **freeze-study** | data + vader + tune(선택) | `freeze_study.csv` | tune 결과 없으면 기본값 사용 | encoder 동결 vs 미세조정 |
| **xai** | **benchmark 체크포인트 필수** | `xai_summary.json`, 케이스별 시각화 | 체크포인트 없으면 에러 (정상) | benchmark 후에만 실행 가능 |
| **dashboard** | 모든 이전 결과 (선택) | `index.html` | `_load_optional_*()` — 없는 건 빈 값 | 부분 결과로도 생성 가능 |

### 1.3 핵심 설계 포인트

**자기 복구 메커니즘:**
- `extract_vader_features()` → 내부에서 `prepare_data()` 호출
- `run_eda()` → 내부에서 `prepare_data()` + `extract_vader_features()` 호출
- `run_benchmark()` → 내부에서 `prepare_data()` 호출, `load_tuned_hyperparams()` 자동 탐색

**캐시 시스템:**
- `data_splits.pkl`, `vader_features.pkl`은 한 번 생성되면 재사용
- `--force` 플래그로 캐시 무시 가능: `./run.sh data --force`

**선택 단계 처리:**
- tune을 안 해도 benchmark가 동작함 → `load_tuned_hyperparams()`가 빈 dict 반환 → 기본 하이퍼파라미터 사용
- dashboard는 존재하는 결과만 표시 → 중간 단계 스킵해도 생성 가능

### 1.4 주의사항

**XAI는 benchmark 없이 실행 불가:**
```bash
# 이렇게 하면 에러
./run.sh data && ./run.sh vader && ./run.sh xai  # ✗ benchmark 체크포인트 없음

# 올바른 순서
./run.sh data && ./run.sh vader && ./run.sh benchmark && ./run.sh xai  # ✓
```

XAI가 `best_models.json`에서 체크포인트 경로를 읽기 때문입니다. 이건 의도된 동작 — 비교할 모델이 없으면 XAI를 돌릴 이유가 없으니까요.

## 2. 팀 분업 전략

### 2.1 권장 구조: 3단계 분업

파이프라인의 특성상 **"공동 → 병렬 → 합류"** 패턴이 자연스럽습니다.

```
┌─────────────────────────────────────────────────────┐
│            Phase 1: 공동 작업 (전체 팀)               │
│  data → vader → eda → 결과 리뷰 & 가설 합의          │
└──────────────────────┬──────────────────────────────┘
                       │
          ┌────────────┴────────────┐
          ▼                         ▼
┌──────────────────┐    ┌────────────────────────────┐
│  팀A: Baseline A  │    │     팀B: Baseline B +       │
│                   │    │     Improved Models         │
│  - TF-IDF + LR   │    │  - BERT-base                │
│  - TF-IDF + SVM  │    │  - BERT+MLP (ablation)      │
│                   │    │  - BERT+VADER               │
│  코드 위치:        │    │  - RoBERTa+VADER            │
│  run_tfidf_       │    │                             │
│  benchmark()      │    │  코드 위치:                   │
│                   │    │  run_transformer_            │
│  결과 해석:        │    │  benchmark()                 │
│  - 전통 ML 한계   │    │                             │
│  - hate F1 낮음   │    │  추가 담당:                   │
│  - BoW의 구조적   │    │  - tune (하이퍼파라미터)       │
│    한계           │    │  - freeze-study              │
└────────┬─────────┘    └────────────┬───────────────┘
         │                           │
         └────────────┬──────────────┘
                      ▼
┌─────────────────────────────────────────────────────┐
│            Phase 3: 합류 (전체 팀)                    │
│  benchmark 결과 비교 → xai → dashboard → 최종 보고서   │
└─────────────────────────────────────────────────────┘
```

### 2.2 Phase 1: 공동 작업 — 데이터 이해

**전체 팀이 함께 수행하는 단계:**

```bash
./run.sh data      # 데이터 다운로드 + 70/10/20 stratified split
./run.sh vader     # VADER 감성 피처 추출
./run.sh eda       # 탐색적 데이터 분석
```

**EDA 결과를 보고 팀 전체가 합의해야 할 사항:**

1. **클래스 분포 확인** → class weighting 전략 적절한지
2. **VADER compound 분포** → hate와 offensive가 실제로 분리되는지 (가설 검증)
3. **어휘 중복도(Jaccard)** → hate/offensive 구분이 구조적으로 어려운 수준인지
4. **타겟 커뮤니티 분포** → 특정 커뮤니티에 hate 집중 여부 (bias 위험)

### 2.3 Phase 2: 병렬 분업

#### 팀A: 전통 ML (Baseline A)

**담당 범위:**
- TF-IDF + Logistic Regression
- TF-IDF + Linear SVM

**이해해야 할 코드:**
- `experiment_core.py` → `run_tfidf_benchmark()` 함수
- TF-IDF 벡터화 (unigram + bigram + trigram)
- scikit-learn 파이프라인 (`Pipeline`, `TfidfVectorizer`, `CalibratedClassifierCV`)

**핵심 역할:**
- 전통 ML의 한계를 정량적으로 보여주는 것이 팀A의 기여
- Davidson et al. (2017): 전체 F1=0.90이지만 hate class F1=0.44 — 이 문제가 우리 데이터에서도 재현되는지 확인
- **결과 해석 포인트:** "BoW 기반 모델이 hate/offensive를 왜 구분하지 못하는가?"

#### 팀B: Transformer (Baseline B + 개선 모델)

**담당 범위:**
- BERT-base (`TransformerCLSClassifier`)
- BERT+MLP (`TransformerMLPClassifier`) — ablation
- BERT+VADER (`HybridSentimentClassifier`)
- RoBERTa+VADER (`HybridSentimentClassifier`)
- Hyperparameter Tuning (`./run.sh tune`)
- Freeze Study (`./run.sh freeze-study`)

**이해해야 할 코드:**
- `experiment_core.py` → 3개 모델 클래스, `run_transformer_benchmark()`, `run_freeze_study()`
- `train_neural_model()` — AdamW + warmup + early stopping 학습 루프

**핵심 역할:**
- BERT-base vs BERT+VADER 성능 차이가 VADER 때문인지 입증
- BERT+MLP ablation으로 "MLP 파라미터 증가 vs VADER 피처" 분리
- Freeze Study로 "VADER만으로 충분한가 vs encoder fine-tuning도 필요한가" 해석

### 2.4 Phase 3: 합류 — 결과 통합

**전체 팀이 함께 수행:**

```bash
./run.sh benchmark     # 전체 모델 벤치마크 (TF-IDF + Transformer 한 번에)
./run.sh freeze-study  # encoder 동결 실험
./run.sh xai           # SHAP + LIME Before/After 비교
./run.sh dashboard     # HTML 대시보드 생성
```

> **참고:** `./run.sh benchmark`는 TF-IDF와 Transformer를 한 번에 다 돌립니다. 실행을 나누는 것이 아니라, **코드 이해와 결과 해석**을 나누는 것입니다.

**합류 시 논의 사항:**
- 벤치마크 결과 통계 검정 (`significance_tests.csv`) — 유의한 차이가 있는가?
- XAI Before/After — VADER 피처의 SHAP 기여도가 실제로 관찰되는가?
- Overlap@5 — SHAP과 LIME이 일관된 해석을 제시하는가?

### 2.5 분업 실무 팁

**실행은 분리하지 않아도 됩니다:**
- `./run.sh all`이 모든 걸 순서대로 돌리므로, 한 사람이 실행하고 결과를 공유해도 됨
- 또는 `./run.sh all --with-tuning`으로 튜닝 포함 전체 실행

**코드 이해 분업이 핵심:**
- 팀A는 `run_tfidf_benchmark()` 함수와 TF-IDF 관련 코드를 깊이 이해
- 팀B는 `TransformerCLSClassifier`, `HybridSentimentClassifier`, 학습 루프를 깊이 이해
- 발표/보고서에서 각자 담당 부분을 설명

## 4. 클래스 불균형 분석 및 대응

### 4.1 실제 분포

```
Normal:     ~38.8%  ████████████████████  (최다)
Hateful:    ~29.5%  ███████████████
Offensive:  ~27.2%  ██████████████        (최소)
Undecided:   ~4.6%  ██                    (제외됨)
```

**판정: 중간 수준(moderate) 불균형**

- 극심한 불균형(1:100 같은)은 아님
- 하지만 Normal이 Offensive보다 약 1.43배 많아서 대응이 필요

### 4.2 이미 적용된 대응 전략

**1. Class Weighting + Label Smoothing (코드에 반영 완료)**

`experiment_core.py`의 학습 루프에서 `CrossEntropyLoss(weight=class_weights, label_smoothing=0.1)`를 사용합니다:

```python
# balanced class weight: hate=1.08, offensive=1.17, normal=0.82
# → offensive에 가장 높은 가중치 → 오분류 시 더 큰 패널티
# label_smoothing=0.1: hate/offensive 경계가 모호한 샘플에서 일반화 향상
```

**2. Stratified Split (코드에 반영 완료)**

`train_test_split(..., stratify=labels)`로 분할하여, train/val/test 각각에서 클래스 비율이 원본과 동일하게 유지됩니다.

**3. Macro F1 사용 (코드에 반영 완료)**

평가 메트릭으로 Accuracy뿐 아니라 **Macro F1** (3클래스 균등 가중 평균)을 주 메트릭으로 사용합니다. Accuracy는 다수 클래스에 편향될 수 있지만, Macro F1은 소수 클래스 성능도 동등하게 반영합니다.

### 4.3 EDA가 이 문제에서 하는 역할

EDA(`./run.sh eda`)로 확인하는 불균형 관련 사항:

| EDA 항목 | 불균형 관련 확인 사항 |
|---------|-------------------|
| 텍스트 길이 분포 | 클래스별 길이 차이 → 모델이 길이에 과의존할 위험 |
| 클래스별 VADER 분포 | hate/offensive/normal 간 compound 분포 분리도 → VADER 가설 타당성 |
| 타겟 커뮤니티 분석 | 특정 커뮤니티에 hate 집중 → 모델 bias 위험 |
| 어휘 중복 (Jaccard) | hate/offensive 어휘 겹침 정도 → 오분류의 구조적 난이도 |

**핵심:** EDA는 "클래스 불균형이 있는지"뿐 아니라 "불균형의 **성격**이 무엇인지"를 파악하는 단계입니다.

### 4.4 추가 대응이 필요한 경우 (현재는 불필요)

만약 EDA에서 극심한 불균형이 발견되면 고려할 수 있는 방법:

| 기법 | 설명 | 현재 필요 여부 |
|------|------|:---:|
| Oversampling (SMOTE 등) | 소수 클래스 합성 | 불필요 (중간 불균형) |
| Undersampling | 다수 클래스 축소 | 불필요 (데이터 손실) |
| Focal Loss | 어려운 샘플에 집중 | 향후 실험 가능 |
| **Class Weighting** | 손실 함수에 역빈도 가중치 | **이미 적용됨** |

## 5. EDA 필요성 정리

### 5.1 "왜 모델 학습 전에 EDA를 해야 하는가?"

| 이유 | 설명 |
|------|------|
| **가설 검증** | "VADER compound가 hate/offensive를 구분한다"는 가설이 데이터에서 실제로 관찰되는지 확인 |
| **불균형 확인** | class weighting 전략의 적절성을 실제 분포로 검증 |
| **난이도 정량화** | hate/offensive 어휘 중복(Jaccard)이 높으면 → 오분류가 모델 문제가 아니라 데이터 구조의 문제일 수 있음 |
| **bias 탐지** | 특정 커뮤니티에 hate가 집중되면 → 모델이 커뮤니티명 자체를 혐오 신호로 학습할 위험 |
| **보고서 근거** | "왜 VADER를 선택했는가?"에 대한 데이터 기반 근거 제공 |

### 5.2 EDA 산출물 목록

```
outputs/reports/eda/
├── text_length_distribution.png   # 클래스별 토큰 수 히스토그램
├── text_length_stats.csv          # 기술통계 (mean, median, std, min, max)
├── vader_by_class.png             # 클래스별 VADER compound/neg boxplot
├── vader_by_class_stats.csv       # VADER 기술통계
├── target_distribution.png        # 타겟 커뮤니티별 빈도 + 클래스 비율
├── target_crosstab.csv            # 커뮤니티 × 클래스 교차표
├── vocabulary_overlap.csv         # 클래스 쌍 간 Jaccard 유사도
├── eda_summary.json               # 전체 요약 (프로그래밍용)
└── eda_report.md                  # 마크다운 리포트 (사람용)
```

### 5.3 EDA 결과 해석 가이드

**VADER compound 분포를 보면:**
- hate의 compound가 offensive보다 확실히 낮다 → VADER 가설 지지 → 파이프라인 설계 타당
- hate와 offensive의 compound가 거의 겹친다 → VADER 가설 약함 → 그래도 실험은 진행하되 결과 해석에 유보 필요

**Jaccard 어휘 중복도를 보면:**
- hate/offensive 중복도 > 0.5 → 구조적으로 구분이 어려운 데이터 → 높은 오분류율이 모델의 실패가 아닐 수 있음
- hate/normal 중복도 < 0.3 → hate/normal은 잘 구분되어야 함 → 여기서 오분류가 많으면 모델 문제
