# 하네스 엔지니어링 적용 사례: HateSpeachStudy

> "모델이 얼마나 똑똑한가"가 아니라, "그 모델을 감싸는 시스템이 얼마나 잘 설계되어 있는가"

---

## 1. 배경: 왜 하네스가 필요했는가

### 1.1 문제 상황

이 프로젝트는 HateXplain 데이터셋 기반 혐오표현 탐지 XAI 파이프라인입니다. AI 코딩 에이전트(Claude Code)를 활용해 개발했는데, 다음과 같은 문제가 반복됐습니다:

| 문제 | 구체적 사례 | 결과 |
|------|-----------|------|
| **Context Drift** | 긴 작업 중 "순환적 프레임워크"라는 잘못된 서술을 계속 생성 | 논문 전체 서술의 일관성 붕괴 |
| **폭주 (Cascading Failure)** | 한 모듈 수정이 README, docs 4개, run.sh에 연쇄 영향 | 문서 간 불일치 누적 |
| **세이브 포인트 부재** | 에러 발생 시 어디로 롤백해야 하는지 불명확 | 수작업으로 diff 추적 |
| **인지 과부하** | AI에게 "전부 고쳐줘"라고 하면 범위가 불분명 | 불필요한 파일까지 수정 |

### 1.2 패러다임 진화 과정

이 프로젝트에서 겪은 AI 활용 방식의 진화:

```
[1단계] 프롬프트 엔지니어링
"BERT+VADER 모델 코드를 짜줘"
→ 문제: 맥락 없이 던지면 기존 코드 스타일과 다른 결과물

[2단계] 컨텍스트 엔지니어링
"experiment_core.py의 TransformerCLSClassifier를 참고해서,
 동일한 패턴으로 BERT+VADER 모델을 만들어줘.
 HybridSentimentClassifier라는 이름으로."
→ 개선: 일관된 코드 생성. 하지만 장기 작업에서 규칙을 잊음

[3단계] 하네스 엔지니어링
"CLAUDE.md를 읽고, progress.md에서 현재 위치를 확인한 뒤,
 architecture.md의 모듈 구조에 맞게 작업하고,
 완료 후 규칙 준수를 자체 검증해줘."
→ 해결: AI가 스스로 계획하고, 실행하고, 검증하는 루프
```

## 2. 적용한 하네스 구조

### 2.1 SSOT (진실의 단일 원천) — 3종 제어 파일

| 파일 | 역할 | 건축 비유 | 내용 |
|------|------|---------|------|
| **CLAUDE.md** | Evaluator (감리사) | 건축 법규 + 품질 기준서 | 코드 규칙, 서술 규칙, git 규칙, 금지 사항 |
| **progress.md** | Planner (건축가의 공정표) | 시공 일정 + 진도표 | 완료 현황, Todo, 세이브 포인트, 알려진 이슈 |
| **architecture.md** | Planner (건축가의 도면) | 설계 도면 + 구조 계산서 | 실행 흐름, 모듈 의존성, 데이터 흐름, 위험 지점 |

### 2.2 4역할 에이전트 분업

```
┌──────────────────────────────────────────────────────────────┐
│  Orchestrator (현장 소장)                                      │
│  run.sh → run_experiments.py                                  │
│  "전체 파이프라인 흐름을 총괄하고, 각 단계를 순서대로 호출"        │
│                                                               │
│  ┌────────────┐   ┌────────────┐   ┌────────────┐            │
│  │  Planner   │──▶│ Generator  │──▶│ Evaluator  │──┐        │
│  │            │   │            │   │            │  │ 실패   │
│  │ progress   │   │ 코드 작성   │   │ CLAUDE.md  │  │        │
│  │ .md 읽기   │   │ 문서 수정   │   │ 규칙 검증   │  │        │
│  │ 작업 계획   │   │ 실험 실행   │   │ 일관성 확인  │  │        │
│  └────────────┘   └────────────┘   └──────┬─────┘  │        │
│                                           │        │        │
│                                        통과?       │        │
│                                     ┌───┴───┐     │        │
│                                     │ Yes   │ No  │        │
│                                     ▼       └─────┘        │
│                            progress.md 업데이트             │
│                            → git commit (세이브 포인트)      │
└──────────────────────────────────────────────────────────────┘
```

### 2.3 생성-검증-재생성 루프 (실제 사례)

**사례: 문서 고도화 작업**

```
[루프 1] Generator: docs/파이프라인_상세분석.md에 EDA 섹션 추가
         Evaluator: 섹션 번호 불일치 발견 (3.1이 아직 남아있음)
         → 재작업

[루프 2] Generator: 전체 섹션 번호 재정리 (4→14)
         Evaluator: 모델 수가 "5개"로 남아있음 (6개여야 함)
         → 재작업

[루프 3] Generator: 모델 테이블 업데이트 + BERT+MLP ablation 설명 추가
         Evaluator: 서술에 "순환적 프레임워크" 잔재 발견
         → 재작업

[루프 4] Generator: "과학적 검증 프레임워크"로 일괄 수정
         Evaluator: 깨진 유니코드 4곳 발견
         → 재작업

[루프 5] Generator: 유니코드 수정
         Evaluator: 통과
         → progress.md 업데이트 → 커밋
```

**핵심:** 한 번에 완벽한 문서를 생성하려 했다면 실패했을 것입니다. 5회 루프를 돌면서 점진적으로 품질을 높였습니다.

## 3. 전략적 제어 수준 설정

### 3.1 3단계 옵션

| 옵션 | 전략적 방향 | 리스크 프로필 | 적용 상황 |
|------|-----------|:----------:|---------|
| **Strict Harness** | 모든 단계를 인간이 승인 | Low: 안전 극대화, 속도 저하 | 파이프라인 흐름 변경, 모델 아키텍처 수정 |
| **Adaptive Harness** | 에러 발생 시에만 개입 | Medium: 속도와 안전의 균형 | 일반 코드 작성, 문서 수정 |
| **Vibe Priority** | 하네스 최소화, AI 자율 | High: 빠른 프로토타이핑, 버그 위험 | 아이디어 스케치, 초기 탐색 |

### 3.2 이 프로젝트의 적용

```
Strict:     [파이프라인 흐름] [모듈 구조] [서술 용어] [논문 인용]
             │
Adaptive:   [코드 구현] [문서 업데이트] [테스트] [리팩토링]
             │
Vibe:       [EDA 시각화 스타일] [주석 톤] [커밋 메시지 단어 선택]
```

**원칙:** 되돌리기 어려운 것일수록 Strict, 쉽게 고칠 수 있는 것일수록 Vibe에 가깝게 설정합니다.

## 4. 세이브 포인트 시스템

### 4.1 왜 필요한가

> "AI에게 일을 시키기 전, '이 작업이 망했을 때 돌아갈 세이브 포인트가 어디인가?'를 먼저 자문해 보세요."

### 4.2 이 프로젝트의 세이브 포인트

| 시점 | git commit | 복원 방법 | 상태 |
|------|-----------|---------|------|
| 하네스 구축 | `839eb1d` | `git checkout 839eb1d` | CLAUDE.md + progress.md + architecture.md 생성 |
| 문서 정리 완료 | `28e82f2` | `git checkout 28e82f2` | 유니코드 수정, 문서 일관성 확보 |
| compat 정리 | `adb9f7b` | `git checkout adb9f7b` | 래퍼 14개 삭제, 깔끔한 구조 |
| 버그 수정 완료 | `2a7bed9` | `git checkout 2a7bed9` | Overlap@5 + XAI + 튜닝 버그 패치 |
| 초기 커밋 | `4bed3a9` | `git checkout 4bed3a9` | 파이프라인 최초 버전 |

### 4.3 운용 규칙

- **의미 있는 작업 단위마다 커밋** → 자동으로 세이브 포인트 생성
- 커밋 메시지에 작업 범위 명시 → 롤백 시 어디까지 되돌릴지 판단 가능
- `progress.md`에 세이브 포인트 테이블 유지 → AI 에이전트가 복원 지점 인식

## 5. 실제 효과 및 교훈

### 5.1 하네스 적용 전후 비교

| 항목 | 하네스 없이 | 하네스 적용 후 |
|------|-----------|-------------|
| **Context Drift** | "순환적 프레임워크" 서술이 4개 문서에 잔존 | CLAUDE.md 규칙으로 일괄 탐지/수정 |
| **문서 간 불일치** | 모델 수가 파일마다 5개/6개/7개로 다름 | Evaluator 루프에서 자동 검증 |
| **롤백** | "어디까지 되돌려야 하지?" 수동 diff | 세이브 포인트 테이블 참조 |
| **작업 범위 제어** | "전부 고쳐줘" → 불필요한 수정까지 | progress.md Todo 기반 범위 한정 |
| **팀 협업** | "이 코드 뭐하는 거야?" | architecture.md로 즉시 파악 |

### 5.2 시행착오: 하네스의 과적합

**문제:** 규칙을 너무 촘촘하게 만들면 AI의 생산성이 떨어집니다.

- 초기에 "모든 변수명을 snake_case로", "모든 함수에 docstring 필수" 등을 규칙에 넣었더니 AI가 기존 코드 스타일과 충돌하며 속도가 급감
- **해결:** "기존 코드 스타일을 따름"이라는 한 줄로 대체. 최소한의 가드레일이 최적

**원칙:** 하네스 규칙은 "이것만은 절대 안 됨"에 집중하고, 나머지는 AI의 판단에 위임합니다.

### 5.3 교훈

1. **git commit이 하네스의 절반이다** — 자주 커밋하는 것만으로도 세이브 포인트 시스템이 완성됨
2. **문서가 곧 제어 장치다** — CLAUDE.md 한 장이 프롬프트 수십 줄을 대체함
3. **루프가 핵심이다** — 한 번에 완벽하게 하려 하지 말고, 생성-검증-재생성 사이클을 신뢰할 것
4. **Strict/Adaptive/Vibe 구분이 속도의 핵심** — 모든 것에 Strict를 걸면 AI의 장점(속도)이 사라짐

## 6. 활용 프롬프트 (Setup Prompt)

이 프로젝트에서 AI 에이전트에게 작업을 시킬 때 사용하는 초기 프롬프트:

```
너는 지금부터 HateSpeachStudy 프로젝트의 하네스 에이전트야.

[Planner] 작업 시작 전:
1. progress.md를 읽고 현재 위치와 다음 할 일을 파악해.
2. architecture.md로 수정 대상의 의존성과 영향 범위를 확인해.

[Generator] 작업 중:
3. CLAUDE.md의 규칙을 준수하면서 작업해.
4. 한 번에 완벽하지 않아도 돼 — 일단 생성하고 검증 단계로 넘어가.

[Evaluator] 작업 후:
5. CLAUDE.md 규칙 위반 여부를 자체 검증해.
6. 문서 간 일관성을 확인해 (모델 수, 파이프라인 흐름, 용어).
7. 검증 실패 시 수정하고 다시 검증해 (루프).

[Orchestrator] 마무리:
8. progress.md를 업데이트해.
9. 논리적 충돌이 발생하면 작업을 멈추고 나에게 승인을 요청해.
```
