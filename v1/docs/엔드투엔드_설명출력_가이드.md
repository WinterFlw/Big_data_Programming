# 엔드투엔드 설명출력 가이드 (v2.2 제안)

> 목적: 현재 v2.1 파이프라인을 크게 훼손하지 않으면서, 모델이 **예측과 함께 최소한의 설명 결과를 직접 출력**하도록 확장하기 위한 팀 공통 가이드라인.

---

## 0. 한 줄 결정

다음 단계에서는 모델 출력 범위를 아래 **4개 항목**으로 제한한다.

1. `혐오표현 여부` (`label`)
2. `핵심 표현` (`key_expression`)
3. `타겟` (`target`)
4. `확신도` (`confidence`)

이 4개만으로도 "무엇을, 왜, 누구에게 향한 말로 판단했는가"를 충분히 보여줄 수 있으며, 추가 설명 변수 확장으로 인한 통제 실패를 피할 수 있다.

---

## 1. 왜 최소 4개만 남기는가

현재 파이프라인은 이미 다음 변수들을 동시에 다루고 있다.

- BERT / RoBERTa
- Attention loss on/off
- VADER on/off
- seed 반복
- 통계 검정
- XAI 4축

여기에 `speech-act`, `context-cues`, `why_hateful`, `counterfactual explanation`까지 한 번에 넣으면:

- 학습 목표가 지나치게 늘어나고
- 평가 지표가 분산되며
- "무엇 때문에 좋아졌는가"를 설명하기 어려워지고
- 팀 단위 구현/검증 난도가 급격히 올라간다.

따라서 이번 단계에서는 **설명 가능한 출력의 최소 핵심 단위**만 채택한다.

---

## 2. 철학 정렬

이 설계는 현재 프로젝트 철학과 정합적이어야 한다.

### 2.1 유지할 원칙

- 모델 입력은 계속 **텍스트만** 사용한다.
- VADER는 텍스트에서 자동 계산되는 파생 피처이므로 유지 가능하다.
- `rationale`, `target`, `source`, `agreement`는 입력이 아니라 **supervision / 분석용 신호**로만 사용한다.
- SHAP/LIME은 계속 **사후 검증(audit layer)** 도구로 유지한다.

### 2.2 바뀌는 점

기존에는 모델이 주로 `label`만 예측하고, 설명은 외부 XAI가 만들어냈다.

앞으로는 모델이 아래를 **직접 출력**한다.

- label
- confidence
- key_expression
- target

즉, `예측 모델 + 설명 출력 번들` 구조가 된다.

### 2.3 중요한 구분

이번 변경은 "**SHAP/LIME을 모델 안에 넣는다**"가 아니다.

정확한 표현은 아래와 같다.

- 모델은 **native explanation output**을 낸다.
- SHAP/LIME은 그 출력과 예측이 **정말 타당한지 감사하는 외부 검증층**이다.

이 구분을 반드시 문서와 발표에서 유지한다.

## 3. 권장 아키텍처 결정

### 3.1 추천안: 공통 설명출력 헤드 + 공통 손실

다음 버전에서는 transformer 계열 모델 전체에 대해 아래 구조를 공통으로 붙인다.

- `classification head`
- `key expression head`
- `target head`

즉, A/B/C/D와 BERT/RoBERTa 차이는 유지하되, **설명출력 능력 자체는 공통 인프라**로 둔다.

이렇게 하면 실험 변수는 크게 늘어나지 않는다.

남는 주요 비교 축:

- `family`: BERT vs RoBERTa
- `attention supervision`: off vs on
- `VADER`: off vs on

설명 출력은 이 3축 비교에서 **상수 인프라**로 취급한다.

### 3.2 왜 이 안이 좋은가

- 각 조건이 동일한 출력 포맷을 가질 수 있다.
- 대시보드 / API / Playground가 단일 스키마로 통일된다.
- 통제 변수 수가 과도하게 늘지 않는다.
- CUDA 이식 후 다회 반복 실험에도 구조가 단순하다.

### 3.3 비권장안

아래 안들은 이번 단계에서는 권장하지 않는다.

- SHAP/LIME 결과 자체를 모델 출력으로 간주
- 자연어 문장 생성형 설명 추가
- speech-act, context cues, why_hateful까지 동시 예측
- 특정 조건만 설명출력 head를 갖고 다른 조건은 미보유

## 4. 출력 스키마

### 4.1 모델 내부 표준 출력

```json
{
  "label_logits": "...",
  "label_probs": "...",
  "confidence": 0.91,
  "key_expression_scores": "...",
  "target_logits": "...",
  "attention_map": "optional"
}
```

### 4.2 사용자/대시보드 노출 출력

```json
{
  "label": "hatespeech",
  "confidence": 0.91,
  "key_expression": "dirty immigrants should leave",
  "target": "immigrants"
}
```

### 4.3 예외 처리 규칙

- `label = normal`이면 `target = null`, `key_expression = null` 허용
- `confidence < threshold_label`이면 결과는 내되 `"uncertain": true` 내부 플래그 허용
- `target score < threshold_target`이면 `target = null`
- `span score < threshold_span`이면 `key_expression = null`

### 4.4 Multi-label target 처리

학습은 multi-label로 하되, 사용자 노출은 아래 중 하나로 제한한다.

- `top-1 target`만 보여주기
- 또는 score가 임계값 이상인 target 중 1개만 보여주기

이번 단계에서는 **top-1 target 단일 출력**을 권장한다.

## 5. 모델 설계 가이드

### 5.1 분류 head

기존과 동일하다.

- 입력: pooled representation (`[CLS]` 또는 equivalent)
- 출력: 3-class logits

### 5.2 핵심 표현 head

권장 구현:

- encoder의 `last_hidden_state` 각 토큰에 대해 `Linear(hidden, 1)` 적용
- sigmoid score로 토큰별 중요도 예측
- 학습 시 `rationale_mask`를 supervision으로 사용

즉, `key_expression`은 attention heuristic이 아니라 **토큰별 native span score**에서 생성한다.

### 5.3 타겟 head

권장 구현:

- pooled hidden 위에 `Linear(hidden, num_targets)`
- multi-label BCE loss
- inference 시 top-1 target 반환

### 5.4 attention output의 역할

attention은 유지하되, 역할을 명확히 제한한다.

- 학습: attention supervision loss
- 분석: heatmap / rollout
- 보조 근거: native explanation audit

단, `attention = explanation`이라고 직접 주장하지 않는다.

## 6. 손실 함수 가이드

권장 손실 구성:

```text
L_total = L_cls + λ_span * L_span + λ_target * L_target + α * L_attn
```

설명:

- `L_cls`: 메인 3-class 분류 손실
- `L_span`: rationale mask 기반 token-level BCE
- `L_target`: multi-label BCE
- `L_attn`: 기존 attention supervision

### 6.1 권장 운영 원칙

- `L_span`, `L_target`는 **모든 transformer 조건에서 공통 적용**
- `L_attn`만 기존 ablation 축에 따라 on/off
- `VADER`만 기존 ablation 축에 따라 on/off

이렇게 해야 설명출력 기능이 실험 교란 변수가 아니라 **공통 인프라**가 된다.

### 6.2 H4 정리 제안

현재 `D_B+Target`은 별도 부가 실험 성격이 강하다.

엔드투엔드 설명출력 버전에서는 두 선택지가 있다.

1. `target head`를 전체 조건 공통 인프라로 승격하고 `D_B+Target`은 종료
2. 기존 v2.1 논문 메시지 유지를 위해 `D_B+Target`을 레거시 부가 실험으로만 보존

팀 운영 측면에서는 **1번이 더 단순하고 통제에 유리**하다.

## 7. 핵심 표현 추출 규칙

모델이 토큰 score를 냈다고 해서 바로 문자열이 되는 것은 아니다.

권장 후처리:

1. special token 제거
2. punctuation-only token 제거
3. threshold 이상 토큰 선택
4. 연속 토큰을 하나의 span으로 병합
5. 최고 점수 contiguous span 1개만 선택

이번 단계에서는 **Top-1 span만 출력**한다.

추가 아이디어:

- 너무 짧으면 양옆 1~2 토큰 확장
- span 길이 상한 8~12 토큰
- `target`과 겹치는 span은 우선 유지

## 11. 팀 분업 권장안

이번 작업은 역할명보다 **수정 파일 ownership**을 먼저 분리하는 것이 중요하다.
같은 파일을 여러 명이 동시에 크게 건드리면 충돌이 많고, 책임도 흐려진다.

권장 원칙:

- 한 파일의 주 소유자(owner)는 1명
- 다른 팀원은 해당 파일에 직접 대형 수정하지 않고 인터페이스 요구만 전달
- 출력 필드명과 null 정책은 공통 계약으로 먼저 고정
- `experiment_core.py`, `dashboard_app.py`는 같은 시간에 여러 명이 크게 수정하지 않음

### 11.0 최상위 ownership 매트릭스

| 영역 | 주 소유자 | 직접 수정 파일 | 최종 책임 산출물 |
|---|---|---|---|
| 모델/학습 | A팀 | `experiment_core.py` | 새 head, 손실, checkpoint, 내부 출력 스키마 |
| 추론/API | B팀 | `dashboard_app.py` | `/api/predict` 응답 JSON, threshold, serialization |
| 대시보드/UI | C팀 | `dashboard_app.py` 프론트 영역 | highlight, badge, confidence UI |
| 평가/XAI | D팀 | `experiment_xai.py` | native-vs-XAI 비교 지표, span/target 평가 |
| 문서/발표 | E팀 | `docs/*.md`, `README.md` | 용어 통일, 메시지, 보고서/발표 반영 |
| 실행/운영 | 팀장 또는 별도 운영 담당 | `run_experiments.py`, `run.sh` | smoke test, 재현 흐름, 통합 검증 |

- 팀원 누구나 같은 설명 문장을 사용할 수 있음

### 11.1 인터페이스 계약

팀 전체가 먼저 아래를 동결한다.

- API fields:
  - `label`
  - `confidence`
  - `key_expression`
  - `target`
- 내부 fields:
  - `label_probs`
  - `key_expression_scores`
  - `target_logits`
- null policy:
  - `normal -> key_expression=null, target=null`
- target policy:
  - top-1 only
- key expression policy:
  - top-1 contiguous span only

### 11.2 추천 협업 순서

1. A팀이 내부 출력 스키마를 먼저 고정
2. B팀이 그 스키마로 API 응답 연결
3. C팀이 같은 응답을 UI에 렌더링
4. D팀이 native output을 읽어 평가/XAI 비교 추가
5. E팀이 이 계약을 문서/발표에 반영
6. 운영 담당이 전체 smoke test 수행

### 11.3 금지된 병렬 작업

- A팀과 B팀이 동시에 `dashboard_app.py` 핵심 추론 로직 수정
- A팀과 D팀이 동시에 `experiment_core.py` 출력 필드명 변경
- E팀이 코드 확정 전 문서에서 필드명 서술 확정

### 11.4 팀장 체크포인트

- 필드명이 코드/API/문서에서 동일한가
- `normal` 케이스 null 정책이 일치하는가
- target이 단일 출력인지 팀 전체가 같은 이해를 갖는가
- XAI가 native output을 대체하는 구조로 오해되고 있지 않은가

## 13. 완료 기준 (Definition of Done)

아래를 만족하면 기능 완료로 본다.

- 모든 transformer 조건이 `label`, `confidence`, `key_expression`, `target` 출력 가능
- `dashboard_app.py` Playground에서 4개가 시각적으로 확인 가능
- `key_expression`가 rationale 기준 token F1으로 정량 평가 가능
- `target`이 top-1 기준으로 정량 평가 가능
- SHAP/LIME은 기존처럼 별도 audit layer로 유지
- benchmark / xai / dashboard 산출물이 깨지지 않음
